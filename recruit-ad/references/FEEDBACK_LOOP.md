# フィードバックループ プロセス定義

> このファイルを読み込むことで、広告分析時に自動的にフィードバックループが回る。

## ディレクトリ構成

```
references/
  hypothesis-driven-analysis.md   # 仮説駆動分析プロトコル
  learned-corrections.md          # 過去の分析失敗・インシデント集（自動追記あり）
  cross-data-sources.md           # クロスデータソース分析ルール
  FEEDBACK_LOOP.md                # このファイル（プロセス定義）
analysis-log/
  YYYY-MM-DD_campaign_summary.md  # 個別分析ログ
```

## プロセス1: 週次分析（/recruit:ad:analyze 実行時に毎回実行）

### 入力
- KPI異常のアラート、または定期レポート作成の指示

### 手順

```
1. references/ 配下の全 .md を読み込む
2. Supabase ad_analysis_knowledge から直近4回分のナレッジを参照
3. 広告プラットフォームからデータ取得（Notion広報運用DB or GAS経由）
4. hypothesis-driven-analysis.md に従い4ステップ分析を実行:
   a. 事実確認（数字を整理、解釈しない）
   b. 仮説の発散（3つ以上、4カテゴリから）
   c. 体系的な検証（cross-data-sources.md のマトリックスに従う）
   d. 反復（新事実→新仮説）
5. learned-corrections.md の既知パターンに該当しないか最終照合
6. 分析結果をNotionに記録 + Slack通知
7. 【重要】分析ログを生成（プロセス2へ）
```

## プロセス2: 分析ログ生成（分析の直後に毎回実行）

分析を1件行うごとに、以下のテンプレートで分析ログを生成する。

### テンプレート

```markdown
# 分析ログ: [対象キャンペーン/事業部]

- **日付**: YYYY-MM-DD
- **対象**: [訪問看護 / デイサービス / 集客]
- **トリガー**: [週次定期 / KPIアラート / 手動依頼]

## 検出した異常

| 指標 | 値 | 前週比 | 目標比 | 深刻度 |
|------|-----|--------|--------|--------|
| ... | ... | ... | ... | 高/中/低 |

## 仮説と検証

| # | 仮説 | カテゴリ | 検証結果 | 確信度 |
|---|------|---------|---------|--------|
| 1 | ... | A/B/C/D | 支持/否定/部分的 | 高/中/低 |
| 2 | ... | ... | ... | ... |
| 3 | ... | ... | ... | ... |

## 使用したデータソース

- [ ] 広告プラットフォーム（Meta/Google）
- [ ] GA4
- [ ] GSC
- [ ] 季節カレンダー
- [ ] サイト変更履歴
- [ ] Supabaseナレッジ

## 結論

[根本原因と推奨アクション]

## 抽出パターン

この分析から得られた汎用的な教訓:

- **パターン名**: [例: デイサービスの週末広告停止]
- **ルール化**: [例: 土日の予算を平日の50%に設定する]
- **該当ファイル**: [例: learned-corrections.md / cross-data-sources.md]
- **新規ルール追加の必要性**: [yes/no]
```

### 保存先
`analysis-log/YYYY-MM-DD_campaign_summary.md`

例: `analysis-log/2026-04-07_home-nursing_cpa-spike.md`

## プロセス3: ナレッジ集約（5件ごとに実行）

analysis-log/ 内のファイル数が前回集約時から5件以上増えたら実行。

### 手順

```
1. analysis-log/ の全ログを読み込む
2. 抽出パターンを集計:
   - 新規ルール追加: ○件
   - 既存ルール強化: ○件
   - パターンなし: ○件
3. 頻出パターン（2回以上）を特定
4. 該当するリファレンスファイルに新ルールを追記:
   - 分析失敗パターン → learned-corrections.md
   - データソース活用 → cross-data-sources.md
   - 分析手法改善 → hypothesis-driven-analysis.md
5. learned-corrections.md の「分析ログからの追加分」セクションを更新
6. Supabase ad_analysis_knowledge にもナレッジを記録
7. 集約結果のサマリーをユーザーに報告
```

## プロセス4: 分析前チェック（分析・レポート・提案の前に毎回実行）

### チェックリスト

```
1. learned-corrections.md の既知パターンを確認済みか
   → 特にCV定義の混同（#1）は毎回照合
2. 分析期間に季節イベントが含まれるか
   → cross-data-sources.md の季節カレンダーで照合
3. 学習期間中のキャンペーンがないか
   → 該当する場合はデータに「⚠️ 学習期間中」フラグを付与
4. 前回分析からの変更点は把握しているか
   → Supabaseナレッジの直近4回分を確認済みか
```

## 運用ルール

- 分析ログは削除しない（蓄積が価値）
- リファレンスの手動編集もOK。自動追記は「分析ログからの追加分」セクションにのみ行う
- リファレンスの変更履歴はGitで管理（claude-skillsリポジトリ）
- 集約サイクルは5件ごと（記事改善の10件より短い。広告分析は週次実行で頻度が低いため）
