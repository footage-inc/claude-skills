# クロスデータソース分析ルール

> 広告プラットフォームだけでは「なぜ」に答えられない。データの外側にある文脈を供給する。

## データソース一覧

### 1. GA4（サイト行動分析）

- **ツール**: `mcp__ga4__analytics_report`, `mcp__ga4__quick_insights`
- **プロパティ**: footage-nursing.jp（ID: 318749283）
- **確認タイミング**: CPA/CVR異常検知時、レポート作成時
- **確認指標**:
  - LP別直帰率（bounce_rate）: 広告LPの問題を検出
  - フォーム到達率・完了率: 応募導線の障害を検出
  - セッション時間: エンゲージメント品質の変化
  - 流入元別セッション数: 広告以外のチャネル変動との相関
  - デバイス別パフォーマンス: モバイル/PCのUI問題

### 2. GSC（検索パフォーマンス）

- **ツール**: `mcp__gsc__query_search_analytics`, `mcp__gsc__get_keyword_trend`, `mcp__gsc__compare_performance`
- **確認タイミング**: オーガニック流入変動時、ブランド認知度の間接評価時
- **確認指標**:
  - 採用関連クエリ（「訪問看護 求人」「PT 求人 名古屋」等）のインプレッション・CTR推移
  - ブランドクエリ（「FOOTAGE 訪問看護」「フッテージ 求人」）の推移: 広告によるブランド認知向上の間接指標
  - 競合動向: 同一クエリでのCTR低下は競合の出現を示唆

### 3. 季節カレンダー

- **参照方法**: 以下のハードコードカレンダーで照合
- **使い方**: 分析期間が季節イベントと重なる場合、仮説に「季節要因」を必ず含める

| 時期 | イベント | 影響 |
|------|---------|------|
| 1月〜3月 | 4月入職に向けた求職ピーク | 応募数↑、CPA↓ |
| 4月前半 | 入職完了・新年度開始 | 求職需要一時低下 |
| 4月後半〜5月 | GW前後 | 求職活動停滞 |
| 6月〜8月 | 通常期 | ベースライン |
| 8月中旬 | お盆 | 1〜2週間の応募減 |
| 9月〜10月 | 下半期入職ピーク | 応募数↑（春ほどではない） |
| 11月〜12月前半 | 通常期 | ベースライン |
| 12月後半〜1月初旬 | 年末年始 | 求職活動停滞 |

### 4. サイト・コンテンツ変更

- **確認方法**: Notion記事DBの最新公開記事、GitHubのコミットログ
- **確認タイミング**: 流入元構成比が変化した場合
- **確認項目**:
  - 採用LP（ランディングページ）の変更有無
  - 採用関連コラム記事の新規公開
  - 応募フォームの変更有無

### 5. 広告管理画面の変更履歴

- **確認方法**: Meta Ads / Google Adsの変更履歴、Notion広報運用DBの実行ログ
- **確認タイミング**: 分析開始時に必ず
- **確認項目**:
  - 予算変更、入札戦略変更、ターゲティング変更
  - クリエイティブの追加・停止
  - 自動最適化による変更

| CPA急上昇 | ✅ 必須 | ✅ LP直帰率・フォーム完了率 | — | ✅ 季節要因確認 | ✅ LP・フォーム変更 |
| CVR急低下 | ✅ 必須 | ✅ フォーム完了率・エラー率 | — | — | ✅ フォーム・LP変更 |

| 応募数減少 | ✅ 必須 | ✅ 流入元別セッション | ✅ 採用KW動向 | ✅ 季節要因確認 | ✅ フォーム障害 |
